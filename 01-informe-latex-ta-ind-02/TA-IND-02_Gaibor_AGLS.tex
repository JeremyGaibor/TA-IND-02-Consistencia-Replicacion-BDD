% ================= RESUMEN =================
\section*{Resumen}
Este informe analiza el modelo de consistencia y replicación utilizado en TiendaTech, una plataforma de comercio electrónico orientada a la venta de componentes informáticos. La implementación actual emplea una aplicación Spring Boot conectada a una única instancia PostgreSQL 17 desplegada mediante Docker Compose. Por tanto, la persistencia se encuentra centralizada y no existe replicación entre nodos. Esta configuración simplifica las transacciones y evita conflictos entre copias, pero convierte a PostgreSQL en un punto único de fallo. El modelo se evalúa mediante los conceptos de consistencia, CAP y PACELC. También se comparan tres alternativas: replicación síncrona primario--respaldo, replicación asíncrona con réplicas de lectura y una estrategia híbrida. Se concluye que el diseño actual es adecuado para un prototipo académico, pero no es óptimo para una plataforma productiva que requiera alta disponibilidad. Se recomienda mantener garantías estrictas para inventario, pedidos y pagos, y utilizar réplicas asíncronas para catálogo, búsquedas y reportes.

\vspace{0.3cm}

\section{Introducción}
TiendaTech es una plataforma de comercio electrónico especializada en componentes informáticos y asistencia para seleccionar productos compatibles. El sistema contempla la gestión de usuarios, catálogo, inventario, carritos, pedidos, ventas y pagos. En este dominio, la persistencia debe evitar resultados contradictorios cuando varias operaciones se ejecutan de forma concurrente. Los sistemas distribuidos modernos deben equilibrar consistencia, disponibilidad, latencia y tolerancia a fallos según las necesidades del negocio \cite{margara2023,taft2020}.

Aunque el PFC plantea una arquitectura basada en servicios, la persistencia realmente implementada utiliza una única instancia PostgreSQL. El archivo Docker Compose contiene el servicio \texttt{app}, correspondiente a la aplicación, y el servicio \texttt{postgres}, encargado del almacenamiento centralizado. No existe un nodo secundario al cual redirigir las operaciones si PostgreSQL deja de responder.

El uso de contenedores o de infraestructura en la nube no convierte automáticamente a una base de datos en distribuida. La replicación requiere mantener copias coordinadas en varios nodos mediante mecanismos de propagación, sincronización o consenso \cite{margara2023,zhou2023}.

El objetivo del informe es caracterizar el modelo real de TiendaTech, analizarlo mediante CAP y PACELC y evaluar alternativas de replicación. La pregunta orientadora es: \textit{¿el modelo de consistencia elegido es el óptimo para el dominio de TiendaTech?}

% ================= MARCO TEÓRICO =================
\section{Marco teórico}

\subsection{Modelos de consistencia}
La consistencia establece las reglas sobre el orden y la visibilidad de las operaciones. No es una propiedad únicamente binaria, debido a que existen modelos con diferentes garantías. Viotti y Vukolić clasifican estas garantías según las relaciones permitidas entre lecturas y escrituras, mientras que estudios recientes comparan cómo los sistemas distribuyen, sincronizan y recuperan sus datos \cite{margara2023,viotti2016}.

La \textbf{linealizabilidad} exige que cada operación parezca ejecutarse instantáneamente entre su inicio y finalización. Una lectura posterior a una escritura confirmada debe observar el valor actualizado. Esta garantía facilita el razonamiento sobre sistemas transaccionales, pero requiere mayor coordinación entre réplicas \cite{taft2020,viotti2016}.

La \textbf{consistencia secuencial} obliga a que todos los procesos observen las operaciones en un mismo orden global, compatible con el orden de cada proceso, aunque no necesariamente con el tiempo real. Es una garantía más débil que la linealizabilidad \cite{margara2023,viotti2016}.

La \textbf{consistencia causal} conserva el orden de las operaciones relacionadas por causa y efecto. Las operaciones concurrentes sin dependencia causal pueden observarse en órdenes diferentes, lo que reduce la coordinación necesaria \cite{margara2023,viotti2016}.

La garantía \textbf{read-your-writes} asegura que un usuario observe sus propias modificaciones posteriores. Esta propiedad evita que, después de actualizar un perfil o añadir un producto al carrito, la misma sesión reciba una versión anterior \cite{margara2023,li2022}.

La \textbf{consistencia eventual} permite diferencias temporales entre réplicas. Si dejan de producirse escrituras, las copias terminan convergiendo. Favorece la disponibilidad y reduce coordinación, pero puede entregar datos atrasados, por lo que no resulta apropiada para todas las operaciones \cite{margara2023,viotti2016}.

En general, los modelos fuertes requieren mayor coordinación, mientras que los débiles pueden reducir latencia y mejorar disponibilidad. Sin embargo, investigaciones recientes proponen protocolos que buscan conservar garantías fuertes reduciendo parte de ese costo \cite{zhou2023,ruan2024}.

\subsection{Teorema CAP}
CAP relaciona consistencia, disponibilidad y tolerancia a particiones. Durante una partición de red, un sistema distribuido no puede garantizar simultáneamente consistencia y disponibilidad \cite{viotti2016,gilbert2002}.

Esto no significa que el sistema elija permanentemente dos características de tres. La decisión aparece cuando los nodos no pueden comunicarse: puede rechazar operaciones para evitar divergencias o continuar respondiendo y aceptar conflictos posteriores \cite{taft2020,gilbert2002}.

En TiendaTech no existe una partición entre réplicas porque solo hay una instancia PostgreSQL. Si la aplicación pierde acceso a ese nodo, no dispone de otra copia operativa y deja de estar disponible.

\subsection{PACELC}
PACELC amplía CAP al considerar que, incluso sin particiones, existe un compromiso entre latencia y consistencia. En presencia de una partición se elige entre disponibilidad y consistencia; en operación normal se equilibra latencia y consistencia \cite{taft2020,abadi2012}.

En una \textbf{replicación síncrona}, el primario espera la confirmación de otro nodo antes de completar la transacción. Esto reduce el riesgo de pérdida de datos, pero incrementa el tiempo de respuesta \cite{taft2020,ruan2024}.

En una \textbf{replicación asíncrona}, el primario confirma primero y las réplicas se actualizan después. La latencia de escritura disminuye, aunque los seguidores pueden devolver datos temporalmente anteriores \cite{zhou2023,ruan2024}.

Para TiendaTech, inventario, pedidos y pagos justifican una mayor coordinación. En cambio, imágenes, descripciones y reportes pueden tolerar breves retrasos \cite{margara2023,li2022}.

\subsection{Estrategias de replicación}
La replicación mantiene copias de los datos en varios nodos para aumentar disponibilidad, tolerancia a fallos y capacidad de lectura. Sin embargo, también obliga a ordenar, propagar y confirmar modificaciones \cite{margara2023,taft2020}.

En la replicación \textbf{primario--seguidor}, un nodo principal recibe las escrituras y transmite los cambios a los secundarios. Los seguidores pueden atender consultas y reemplazar al primario si existe un mecanismo de promoción y \textit{failover} \cite{margara2023,taft2020}.

La replicación \textbf{multi-líder} permite que varios nodos acepten escrituras. Puede reducir latencia geográfica, pero introduce conflictos concurrentes. GeoGauss emplea una arquitectura multi-maestro con control de concurrencia y procesamiento por épocas para conservar consistencia fuerte \cite{margara2023,zhou2023}.

En la replicación \textbf{sin líder}, las operaciones se distribuyen entre varias copias mediante cantidades mínimas de nodos para leer y escribir. Este enfoque favorece la disponibilidad, pero requiere gestionar versiones concurrentes y reconciliación \cite{margara2023,viotti2016}.

Dynamo constituye un caso representativo de replicación sin líder. Distribuye datos entre varios nodos, utiliza lecturas y escrituras configurables y permite reconciliar versiones concurrentes posteriormente. Su diseño prioriza disponibilidad y tolerancia a fallos, aceptando consistencia eventual \cite{decandia2007}.

La replicación puede ser síncrona o asíncrona. La síncrona protege mejor las operaciones confirmadas, pero añade latencia. La asíncrona reduce el tiempo de escritura, aunque puede entregar información atrasada en las réplicas \cite{taft2020,ruan2024}.

\subsection{Estado del arte}
CockroachDB combina particionamiento, replicación, consenso y control transaccional para proporcionar transacciones SQL consistentes sobre infraestructura distribuida. Su diseño demuestra que la tolerancia a fallos requiere coordinación y recuperación, no solamente copias de datos \cite{margara2023,taft2020}.

GeoGauss propone una arquitectura geo-replicada multi-maestro para cargas OLTP. Su diseño integra replicación y control de concurrencia para reducir coordinación sin renunciar a garantías fuertes \cite{margara2023,zhou2023}.

PolyBase analiza cómo la ubicación de los líderes y grupos de consenso afecta la latencia. La propuesta reasigna datos para acercarlos a las regiones desde las que reciben más solicitudes \cite{taft2020,ruan2024}.

Stabilizer plantea niveles de consistencia definidos por la aplicación. Este enfoque resulta útil cuando distintos módulos requieren garantías diferentes, como ocurre entre inventario y reportes en TiendaTech \cite{margara2023,li2022}.

Dynamo y Spanner representan extremos distintos. Dynamo prioriza disponibilidad mediante replicación sin líder y consistencia eventual \cite{decandia2007}. Spanner proporciona consistencia externa mediante replicación coordinada y sincronización temporal \cite{corbett2013}.

La comparación muestra que no existe una estrategia universalmente superior. Dynamo acepta divergencias temporales para seguir respondiendo, mientras Spanner asume mayor coordinación para ofrecer un orden global más estricto \cite{decandia2007,corbett2013}.

Para TiendaTech, una arquitectura completamente eventual sería riesgosa para inventario y pagos, mientras que una infraestructura similar a Spanner sería excesiva para el alcance actual. La solución debe aplicar garantías diferentes según la criticidad de los datos \cite{margara2023,taft2020}.

\section{Análisis del modelo implementado en TiendaTech}

\subsection{Caracterización de la implementación}
La implementación utiliza una única instancia PostgreSQL 17. El Docker Compose define el contenedor \texttt{tiendatech\_postgres}, expuesto en el puerto 5432, y el contenedor \texttt{tiendatech\_app}, expuesto en el puerto 8080.

La aplicación depende directamente de PostgreSQL. No existen configuraciones de réplica, seguidores, quórum, consenso o promoción automática.

El volumen \texttt{postgres\_data} conserva los archivos de PostgreSQL cuando el contenedor se reinicia o se vuelve a crear. Sin embargo, no constituye una réplica porque no mantiene una segunda instancia operativa.

La caracterización correcta es la siguiente:

\begin{quote}
\textit{TiendaTech utiliza una base de datos PostgreSQL centralizada, con consistencia transaccional local y sin una estrategia de replicación implementada.}
\end{quote}

\subsection{Topología de persistencia}
La Figura~\ref{fig:topologia} presenta la topología realmente implementada. El usuario accede a la aplicación, esta realiza operaciones mediante JDBC/SQL y PostgreSQL conserva sus archivos en el volumen Docker.

\begin{figure}[H]

    \caption{Topología actual de persistencia de TiendaTech. Fuente: elaboración propia.}

Como se observa en la Figura~\ref{fig:topologia}, PostgreSQL constituye el único nodo de datos. No existen nodos secundarios, replicación ni \textit{failover} automático.

El volumen persistente protege frente a la recreación del contenedor, pero sigue dependiendo del mismo servidor. Si se pierde la máquina, el almacenamiento o la instancia PostgreSQL, la aplicación no puede continuar utilizando otra copia.

\subsection{Modelo de consistencia identificado}
TiendaTech obtiene consistencia transaccional local porque todas las operaciones se ejecutan sobre la misma instancia PostgreSQL. No existe retraso de propagación entre réplicas.

No sería correcto afirmar que el sistema implementa linealizabilidad distribuida, ya que no existen varios nodos coordinados. Tampoco puede asegurarse que todas las transacciones sean serializables sin revisar el nivel de aislamiento configurado.

La ventaja del nodo único es la ausencia de conflictos entre copias. La limitación es que la consistencia se obtiene sin redundancia y toda la aplicación depende de la disponibilidad de PostgreSQL.

\subsection{Estrategia de replicación actual}
La estrategia de replicación implementada es \textbf{inexistente}. Esta conclusión se basa en el Docker Compose, donde solamente aparece un contenedor PostgreSQL:

\begin{lstlisting}[caption={Fragmento representativo del Docker Compose de TiendaTech}, label={lst:compose}]
services:
  tiendatech_app:
    build: .
    ports:
      - "8080:8080"
    depends_on:
      - tiendatech_postgres
    restart: always

  tiendatech_postgres:
    image: postgres:17
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: always

volumes:
  postgres_data:
\end{lstlisting}

La ausencia de replicación reduce complejidad, consumo de recursos y esfuerzo de administración. Esto resulta razonable durante una etapa académica inicial.

Sin embargo, PostgreSQL se convierte en un punto único de fallo. La opción \texttt{restart: always} reinicia el mismo contenedor, pero no reemplaza el nodo cuando falla el servidor o el almacenamiento.

\subsection{Requisitos del dominio}
El inventario requiere garantías estrictas. Si dos usuarios intentan comprar la última unidad, la verificación y disminución del \textit{stock} deben realizarse de forma coordinada \cite{taft2020,zhou2023}.

Los pedidos y ventas necesitan atomicidad para evitar registros parciales. Los sistemas SQL distribuidos utilizan protocolos transaccionales para conservar coherencia entre datos replicados o particionados \cite{margara2023,taft2020}.

Los pagos requieren idempotencia y almacenamiento consistente para evitar cobros duplicados. Los usuarios, roles y credenciales también necesitan que los cambios sean visibles rápidamente \cite{margara2023,li2022}.

El catálogo, imágenes y reportes pueden tolerar retrasos breves. Mostrar temporalmente una descripción anterior tiene menor impacto que confirmar una venta sin \textit{stock} \cite{margara2023,ruan2024}.

\subsection{Análisis mediante CAP}
En la topología actual no existe una partición entre réplicas. Si la aplicación pierde acceso a PostgreSQL, el servicio de datos deja de estar disponible.

El diseño evita divergencias entre copias, pero no ofrece tolerancia frente al fallo del nodo. Si se incorporaran réplicas, inventario y pagos deberían priorizar consistencia, mientras catálogo y reportes podrían mantener disponibilidad mediante lecturas atrasadas \cite{margara2023,gilbert2002}.

\subsection{Análisis mediante PACELC}
Durante el funcionamiento normal, TiendaTech no espera confirmaciones remotas, lo que reduce la latencia.

Una réplica síncrona aumentaría el tiempo de confirmación, mientras una réplica asíncrona podría entregar valores anteriores. El costo dependería del protocolo, la distancia y la ubicación del líder \cite{taft2020,ruan2024}.

La decisión más adecuada es aplicar garantías fuertes a procesos críticos y permitir menor coordinación en consultas que toleren retrasos \cite{margara2023,li2022}.

% ================= EVALUACIÓN DE ALTERNATIVAS =================
\section{Evaluación de alternativas}

\subsection{Primario con respaldo síncrono}
Esta alternativa mantiene un PostgreSQL primario y agrega un respaldo. Las transacciones críticas se confirman después de que el secundario recibe el cambio.

Reduce el riesgo de pérdida de operaciones confirmadas y resulta apropiada para pedidos, pagos e inventario. Su desventaja es el aumento de latencia y la necesidad de implementar promoción y redirección de conexiones.

\subsection{Primario con réplica asíncrona}
Esta alternativa conserva un primario para escrituras y añade seguidores para lecturas.

Permite distribuir consultas de catálogo, imágenes y reportes. Sin embargo, las réplicas pueden mostrar información atrasada, por lo que no deberían utilizarse para confirmar \textit{stock}.

\subsection{Estrategia híbrida}
La estrategia híbrida combina:

\begin{itemize}
    \item primario para escrituras;
    \item respaldo síncrono para operaciones críticas;
    \item réplica asíncrona para catálogo y reportes;
    \item \textit{failover};
    \item monitoreo;
    \item copias de seguridad.
\end{itemize}

Ofrece el mejor equilibrio, aunque incrementa la complejidad de administración y enrutamiento de consultas.

% ================= DISCUSIÓN CRÍTICA =================
\section{Discusión crítica}

\subsection{¿Es óptimo el modelo actual?}
El modelo actual es adecuado para una etapa académica porque simplifica el desarrollo y evita problemas de sincronización.

Sin embargo, no es óptimo para producción. PostgreSQL constituye un punto único de fallo y todas las operaciones dependen de su disponibilidad.

El volumen Docker conserva archivos, pero no proporciona una segunda instancia. \texttt{restart: always} tampoco ejecuta \textit{failover} hacia otro servidor.

Por tanto:

\begin{quote}
\textit{El modelo centralizado es suficiente para el prototipo actual, pero no es óptimo para una versión productiva que requiera alta disponibilidad, escalabilidad y tolerancia a fallos.}
\end{quote}

\subsection{Trade-offs}
El primer compromiso es simplicidad frente a disponibilidad. El segundo es latencia frente a durabilidad. El tercero es disponibilidad frente a actualidad de lectura. El cuarto es consistencia uniforme frente a consistencia diferenciada \cite{margara2023,ruan2024}.

\subsection{Limitaciones}
Las principales limitaciones son:

\begin{itemize}
    \item Punto único de fallo.
    \item Ausencia de \textit{failover}.
    \item Escalabilidad limitada.
    \item Dependencia del mismo servidor y volumen.
    \item Interrupciones durante mantenimiento.
    \item Falta de redundancia operativa.
\end{itemize}

\subsection{Mitigaciones}
Se propone:

\begin{enumerate}
    \item Mantener transacciones para controlar el \textit{stock}.
    \item Incorporar una réplica asíncrona.
    \item Evaluar respaldo síncrono para pagos y pedidos.
    \item Mantener copias de seguridad externas.
    \item Monitorear disponibilidad y retraso.
    \item Implementar idempotencia.
    \item Probar \textit{failover} y restauración.
\end{enumerate}

\subsection{Propuesta recomendada}
La propuesta recomendada es una evolución híbrida. Primero se añadiría una réplica asíncrona para catálogo, reportes y recuperación. Posteriormente se incorporaría un respaldo síncrono para inventario, pedidos y pagos.

Las consultas críticas permanecerían en el primario, mientras las consultas tolerantes a retrasos se dirigirían a las réplicas.

% ================= CONCLUSIONES =================
\section{Conclusiones}
TiendaTech utiliza una única instancia PostgreSQL y no posee replicación. Su modelo corresponde a persistencia centralizada con consistencia transaccional local.

La solución es adecuada para un prototipo, pero presenta un punto único de fallo y no garantiza continuidad operativa.

CAP muestra que, al perder el acceso a PostgreSQL, la aplicación deja de estar disponible. PACELC evidencia que la baja latencia actual se obtiene al no coordinar réplicas.

La replicación síncrona es adecuada para inventario, pedidos y pagos. La replicación asíncrona resulta útil para catálogo y reportes.

El modelo actual no es óptimo para producción. La alternativa recomendada es una estrategia híbrida que combine consistencia fuerte para procesos críticos y réplicas asíncronas para lecturas no críticas.
